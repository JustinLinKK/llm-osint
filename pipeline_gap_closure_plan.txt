Implementation Plan to Close the Gap Between Pipeline Output and Benchmark
(Prepared for Coding Agent)

Context / Inputs
- Current pipeline report: pipeline_result.md
- Benchmark report: Benchmark.txt
- Future tool planning spec: Tools-future-planning.txt
- Current system prompt(s) for workers/planner: system_prompt pipeline.txt

High-level diagnosis (why benchmark wins)
1) Current pipeline behaves like “snippet -> narrative summarizer”
   - Weak entity canonicalization, weak corroboration, weak timeline grounding.
2) Stop condition is not coverage-led
   - Planner can terminate without satisfying required coverage categories.
3) Missing deterministic follow-up chains
   - Benchmark runs predictable pivots: academic -> thesis/official docs -> coauthors -> clusters, etc.
4) Graph is underutilized
   - Even with a graph-construction worker, downstream reporting does not consistently use typed entities/relations.

Goal
Shift pipeline from summarizer to coverage-led, graph-driven OSINT workflow:
- Hard-gated coverage ledger
- Deterministic follow-up rules
- Typed entity/relationship model
- Graph-driven reporting sections (timeline, collaboration clusters, limitations)

PHASE 0 — Evaluation Harness (build first)
Objective: measure progress and prevent regressions.

0.1 Add a benchmark diff script: benchmark_diff.py
Inputs:
- pipeline_report.md (latest run)
- benchmark.txt (ground truth baseline)
Outputs:
- Section coverage diff
- Entity-type coverage diff (Person/Org/Institution/Publication/Domain/Document/Conference/etc.)
- Relationship-type coverage diff (WORKS_AT, AFFILIATED_WITH, COAUTHORED_WITH, HAS_EMAIL, etc.)
- “Hard anchors” presence:
  - institutional email domains
  - stable profile IDs (OpenReview/ORCID/Scholar IDs)
  - official docs (thesis PDFs / institutional pages)
  - concrete dates (timeline milestones)
  - collaborator clustering presence

0.2 Add a rubric JSON aligned to coverage categories:
{
  "identity": 0,
  "aliases": 0,
  "academic_history": 0,
  "employment_history": 0,
  "publications": 0,
  "relationships": 0,
  "contacts": 0,
  "code_presence": 0,
  "business_roles": 0,
  "archived_history": 0
}

0.3 Scoring heuristics (simple, deterministic)
Examples:
- identity=1 if Person node exists + ≥1 stable identifier (profile URL / scholar ID / ORCID)
- contacts=1 if Email/ContactPage exists and linked via HAS_EMAIL/HAS_CONTACT_PAGE
- relationships=1 if COAUTHORED_WITH/WORKS_AT/ADVISED_BY/MEMBER_OF exists
- archived_history=1 if Wayback snapshot entity exists

Deliverable: “coverage scorecard” printed every run.

PHASE 1 — Coverage Ledger + Hard Stop Logic (highest leverage)
Objective: prevent premature termination; enforce completeness.

1.1 Implement CoverageLedger
File: coverage_ledger.py
@dataclass
class CoverageLedger:
    identity: bool = False
    aliases: bool = False
    history: bool = False
    contacts: bool = False
    relationships: bool = False
    code_presence: bool = False
    business_roles: bool = False
    academic_profile: bool = False
    archived_history: bool = False

1.2 Deterministic ledger update rules
Update ledger from structured outputs of GRAPH_CONSTRUCTION worker (not narrative text).
Examples:
- identity=True if Person + stable identifier exists
- aliases=True if Person has alt_names/handles
- contacts=True if Email/Phone/ContactPage entity exists and is linked
- relationships=True if any relation edge among people/orgs exists (coauthor/employer/advisor)
- archived_history=True if ArchivedPage entity exists

1.3 Planner gating (enforce in loop)
Planner MUST NOT stop until:
- required ledger fields satisfied OR
- max rounds reached, and report includes explicit “unresolved coverage” list.

Deliverable: termination depends on ledger, not planner intuition.

PHASE 2 — Deterministic Follow-up Engine (rules > vibes)
Objective: benchmark-grade pivoting and corroboration.

2.1 Implement followup_rules.py (event-driven)
Trigger after each iteration:
Inputs:
- new entities/relations
- current ledger deficits

Core rule set (minimum viable)
A) Academic rule:
IF publication signal exists (arXiv author page / ≥N publications)
THEN run (tiered):
- semantic_scholar_search
- dblp_author_search
- google_scholar_profile_search
- orcid_search
Optional: conference_profile_search when venue list exists

B) Institution rule:
IF Institution entity exists (e.g., UCSD, UIUC)
THEN run:
- institution_directory_search
Optional: advisor_lineage_search when advisor hints exist

C) Coauthor rule:
IF ≥3 unique Person nodes linked by COAUTHORED_WITH
THEN:
- coauthor_graph_search
- build collaborator clusters (Phase 4)

D) Contact rule:
IF Domain/Website exists AND ledger.contacts=False
THEN:
- contact_page_extractor
- email_pattern_inference (only when name+domain prerequisites satisfied)

E) Archive rule:
IF strong profile URL exists AND ledger.archived_history=False
THEN:
- wayback_fetch_url
- historical_bio_diff

2.2 Tool tiering enforcement
Respect your existing system prompt guidance:
- prioritize general web search first (e.g., Tavily)
- then specialized tools (scholar, dblp, openreview, etc.)
- avoid high-risk categories unless prerequisites met

Deliverable: deterministic follow-ups fill ledger deficits automatically.

PHASE 3 — Tool Output Normalization & Pivot Preservation
Objective: prevent “summary-only” outputs; preserve identifiers for graph.

3.1 Add validator for worker summaries
Reject or re-run summarization if required pivot fields are missing when present:
- URLs
- stable IDs (arXiv ID, DOI, Scholar/ORCID IDs)
- institutions/orgs
- dates

3.2 Treat PDFs/Thesis as first-class entities
If summary contains a .pdf or thesis repository URL:
- emit Document entity
- link via MENTIONS/AFFILIATED_WITH
This enables benchmark-like extraction from thesis acknowledgements and official docs.

Deliverable: official documents become anchor nodes and trigger follow-ups.

PHASE 4 — Graph Model Upgrades (enable benchmark-grade synthesis)
Objective: typed entities/relations + collaboration clustering.

4.1 Implement canonical entity types (minimum set)
- Person
- Organization
- Institution
- Publication
- Document
- Conference
- Domain
- Website
- Handle
- Email
- Location

4.2 Implement canonical relations (minimum set)
- HAS_HANDLE
- HAS_EMAIL
- HAS_PROFILE
- WORKS_AT
- AFFILIATED_WITH
- STUDIED_AT
- PUBLISHED
- COAUTHORED_WITH
- ADVISED_BY
- MEMBER_OF
- LOCATED_IN
- HAS_DOMAIN
- APPEARS_IN_ARCHIVE

4.3 Collaborator clustering module (benchmark feature)
Build coauthor adjacency from COAUTHORED_WITH edges.
Compute:
- connected components (MVP) OR community detection (optional)
Label clusters: Collaboration Group A/B/C…
Feed clusters into reporting as structured evidence.

Deliverable: output includes collaborator groupings similar to benchmark.

PHASE 5 — Reporting Changes (graph-driven, claim-first)
Objective: convert graph into benchmark-like narrative with stronger grounding.

5.1 Entity-aware retrieval for each section
For sections like “Academic trajectory” or “Affiliations”:
- pull directly from graph: institutions, documents, profile URLs, dates
- supplement with vector retrieval only as needed

5.2 Mandatory timeline section
Collect all dated facts (paper years, join dates, degree dates, domain reg dates).
Sort chronologically.
Render as timeline claims + citations.

5.3 Auto-generate “Methodological limitations”
From ledger deficits + negative search receipts:
- “No public phone found”
- “No GitHub identity match”
- “Company identity unresolved; filing tools not executed due to missing jurisdiction/identifier”
This should be explicit, not implied.

Deliverable: report contains (a) timeline, (b) collaboration clusters, (c) limitations.

Milestone 3 (Benchmark-style reporting)

- Graph-driven collaboration clusters section

Acceptance Tests (definition of “gap closed”)
- Alias resolution: canonical vs alternate names with cross-platform evidence
- Academic trajectory: dated milestones + corroborated institutions
- Collaboration: coauthor clusters generated from graph edges
- Documents: at least one official doc node when available (thesis/institution page)
- Limitations: explicit unresolved ledger categories + why not resolved

Notes for Integration with Existing System Prompt
- Do NOT rewrite your prompts unless necessary. Most of the improvement is code-level:
  - enforce ledger gating
  - enforce deterministic follow-ups
  - enforce typed graph usage in reporting
- Treat GRAPH_CONSTRUCTION worker output as the authoritative structure layer.
